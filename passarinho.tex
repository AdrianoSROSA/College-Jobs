\section{Introdução}
A diferença entre um mau e um bom programador é o entendimento sobre o que esta fazendo.
Os computadores causaram uma importante mudança social. A razão é que eles nos permitem fazer mais trabalho com menos pessoas.
O problama é que os computadores quebram.

\subsection{O que há de errado com os computadores?}
Para o software quebrar há apenas uma razão: má programação.
O computador é provavelmente o dispositivo mais complexo que podemos produzir em uma maquina hoje.
Um programa escrito em um computador é igualmente complexo.
A complexidade de um programa pode particularmente confundir, porque não há nada em que color suas maõs.
Assim, programar tem que se tornar o ato de reduzir complexidade à simplicidade.
Um "mau programador" é apenas alguém que falha em reduzir a complexidade.
Então, um "bom programador" deve fazer tudo que estiver ao seu alcance para tornar o que escreve o mais simples possivel para outros programadores.

\subsection{O que é um programa, realmente?}
Um programa de computador é essas duas coisas:\\
As instruções que o programador escreve e as ações que o computador executa.


\section{A ciência ausente}
Projeto de software é:\\
Elaborar um plano para uma criação;\\
Um plano que foi elaborado para uma criação que ainda não foi construida;\\
O plano que uma criação existente segue.
Qualquer coisa que envolve a arquitetura de seu sistema de software ou as decisões técnicas que você toma enquanto cria o sistema se enquadra na categoria de "projeto de software".

\subsection{Todo programador é um projetista}
O desenvolvedor líder esta encarregado de projetar a arquitetura geral do programa inteiro.\\
Os programadores seniores estão encarregados de projetar suas próprias grandes áreas.\\
E os programadores juniors estão encarregados de projetar suas partes do programa.\\
Todos aqueles que escrevem software são projetistas.
Cada pessoa em uma equipe de software é responsavel por se certificar de que seu codigo esta bem projetado.
Ninguém que esteja escrevendo código para um projeto de software pode ignorar projeto de software, em qualquer nivel.
Todos os desenvolvedores devem ter a autoridade de tomar boas decisões de projeto em suas próprias áreas.
Se eles tomarem decisões erradas ou medíocres, essas devem ser canceladas por um superior.
A responsabilidade pelo projeto do código deve permanecer com as pessoas que estão realmente trabalhando nele.
Um projetista deve sempre estar disposto a ouvir sugestões e feedback.
Mas após considerar todos os detalhes, qualquer decisão dada deve ser tomada por uma pessoa, não por um grupo de pessoas.

\subsection{A ciência de projeto de software}
Para que um assunto seja uma ciência, ele tem que passar por certos testes:\\
Uma ciência deve conter conhecimento que for coletado.\\
Esse conhecimento deve ter algum tipo de organização.\\
Uma ciência deve conter verdades gerais ou leis básicas.\\
Uma ciência deve lhe dizer como fazer algo no universo físico.\\
Normalmente, uma ciência é descoberta e comprovada por meio de método científico.\\
No mundo do software, falta a parte mais importante: leis claramente declaradas - verdades inabaláveis que nunca falharão.\\
Definições: lhe dizem o que algo é e como você o usaria.\\
Fatos: são apenas declarações verdadeiras sobre algo.\\
Regras: elas normalmente lhe dizem se dve ou não tomar alguma decisão.\\
Leis: são fatos que serão sempre verdadeiros e abrangem uma ampla área do conhecimento.\\
Projeto de software pode ser uma ciência.
